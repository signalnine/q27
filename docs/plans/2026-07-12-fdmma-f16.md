# fdmma-f16: fp16-MMA verify attention for sm_86 (and fp16-KV everywhere)

2026-07-12. Goal: beat llama.cpp wall clock on 3090 thunderdome tasks.
Sizing (measured, cctx2 replay @27K, turbo3, W8 server, 3090): verify wall
= 53% of decode (phv 1456-1481ms of dec_ms 2749-2835); the 5090 A/B priced
mma-vs-fd2 at 4.4 ms/rnd of a 23-27ms round. Est. prize on 3090: 15-20%
decode -> 70.0 median moves to ~80-84 = llama parity or better (llama
80.7). Bonus: fp16-KV gets an mma verify leg on every arch (it has none).

## Design

New operand mode for k_attn_fdmma: half fragments via m16n8k16.f16 MMA
(the prefill k_attn_prefill_mma already uses this shape on sm_86 -- lift
its fragment/ldmatrix conventions).

- Separate kernel k_attn_fdmma_h16<W, T3> (not a template axis on the
  e4m3 kernel -- the f16 path is structurally SIMPLER): prefill's f16
  idioms mean NO s_P relayout (S fragments convert straight to PV
  A-frags in registers: pa[2][4] via __floats2half2_rn, "subtile pair
  (2n,2n+1) supplies a0..a3", prefill.cu:1144-1153) and NO s_vt
  transpose (ldm_x2_trans reads natural [key][dim] V rows,
  prefill.cu:1155-1165). PV runs 2x k16 steps per n. smem = s_q + s_k +
  s_v + geo only, ~59KB at W8 STAGES=1 -- 2 CTAs/SM fits sm_86's 99KB.
  QK^T: mma_f16 m16n8k16, HD/16 iters, frags at kb+tg*2/+8
  (prefill.cu:1070-1088). Geometry/split/mask/online-softmax/epilogue
  copied from the e4m3 fdmma kernel verbatim.
  Helpers to share: mma_f16, ldm_x2_trans, h2u (defined in prefill.cu
  ~840-900; lift into a shared header or duplicate).
- smem doubles vs e4m3: W8 STAGES=1 ~82KB -- fits the 99KB sm_86 cap.
  W>8 H16 does NOT fit on sm_86; instantiate H16 for W 4..8 only (the
  3090 is a W_MAX=8 build; 5090 fp16-KV use can extend W12 later under
  its 227KB cap if wanted).
- Tile loads by KV format, all landing in half tiles: turbo3 ->
  turbo3_stage8_h2 (exists, prefill-tested); fp16 -> raw half copy;
  fp8 -> kv2h. cp.async stays off for turbo3 (blocking expand, 2nd CTA
  hides it, same as the e4m3 T3 leg).
- Dispatch: attn_decode3 mma gate currently requires image sm_89+. New
  rule: sm_89+ -> e4m3 kernels (unchanged); sm_80..88 + Q27_FD=mma ->
  H16 kernels for kvk in {fp16, fp8, turbo3}, W 4..8. Server profile:
  set Q27_FD=mma on sm_86 too once gated.
- Q staging: fp32 -> half (vs fp8's e4m3 cast) -- H16 is STRICTLY less
  lossy than the shipped e4m3 fdmma (Q keeps 10 mantissa bits, P too).
  Tolerance class bounded by the e4m3 control in test_attn_fdmma_turbo3.

## Gates

1. test_kernels: extend fdmma tests to H16 legs -- control fallthrough,
   engaged, rel vs fd2 gated RELATIVE to the e4m3 fdmma control floor
   (H16 should come in BELOW it), determinism. Runs on BOTH arches (no
   sm_89 skip for H16).
2. Canonical: fp16/fp8/v1/fp8mma legs bitwise vs pre-change binary
   (5090); CLI canonical untouched (mma is serving-only).
3. Sanitizer filtered+capped on the new kernel (both arches).
4. 3090 accept A/B (accept_kv_ab, turbo3 leg now mma): ms/rnd and tps
   vs yesterday's fd2 numbers.
5. The headline: thunderdome T8 x3 on 3090 q27+turbo3+mma vs yesterday's
   llama 80.7 t/s / 242-479s walls. Beat wall medians.

## Non-goals

W>8 H16 on sm_86 (smem), H16 as 5090 default (e4m3 is faster there --
K-loop half as long), turbo2/turbo4 legs.

## Correction (pre-implementation smem math)

H16 W8 smem = s_q 48x264x2 (25.3K) + K/V tiles 32x264x2 each: ~59KB
single-buffered -- TWO CTAs need 118KB > sm_86's 99KB. So the 2-CTA
STAGES=1 trick is f16-impossible at W8; the H16 kernel uses the shipped
STAGES=2 shape instead (1 CTA, double-buffered K/V via cp.async ping-pong,
25.3 + 2x16.9 + 2x16.9 = 92.9KB, fits). Launcher computes ns =
SMs*1/kv_heads (one full wave at 1 CTA/SM; the e4m3 rule of SMs*2 assumed
2 resident). turbo3 leg: blocking expand into the CURRENT buffer (no
cp.async), next-tile prefetch degenerates to nothing -- acceptable first
cut, profile later.
