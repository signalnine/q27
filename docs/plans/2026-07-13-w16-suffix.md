# W16: raise the suffix verify cap 12 -> 16

> **RESOLVED 2026-07-13 (same day): BUILT, GATED, MEASURED -- NO-GO.**
> The cap-binding premise reproduces (95% of fires pin at 12; still 94% pin at
> 16 after widening, so W16 delivers exactly the predicted +33% tokens/fire) --
> but throughput FALLS 434.5 -> 368.5 t/s, because the wide round costs more
> than proportionally. Per-token cost bottoms at W12 (2.103 ms) and rises past
> it. Tokens-per-fire was never the throughput variable. Q27_W_MAX stays 12; no
> w16 build target ships.
>
> The plumbing shipped anyway (W_PLUMB=16, loop-built lane lists, literals
> derived) because the refactor killed a bug class and an audit found a real
> nullptr in refinish_round on the way. And building it surfaced the ACTUAL
> bottleneck the plan was blind to: the verify GEMV was spilling registers
> under a 3-CTA launch_bounds pin. Retiering widths >=10 to a 2-CTA pin is
> worth **+18% t/s on the SHIPPED width 12** (367 -> 434), no widening needed.
>
> Two cliffs, both at W>=14, explain the loss: fdmma's s_q crosses 80 -> 96
> rows so its 2-CTA occupancy dies (52.1KB x 2 > the 100KB/SM limit), and the
> gemv marginal stays superlinear even retiered. W=13 is the last cheap width.
> W16 reopens ONLY behind the mma16 NT=16 GEMM verify pivot (flat in W).
>
> Full numbers: docs/BUILDLOG.md, entry of the same date.

## Original plan (kept for the record -- its "honest expectation" was right about the mechanism and wrong about the payoff)

2026-07-13. Driven by the external fork-maintainer A/B (BUILDLOG same
date): on file re-emission, llama's unbounded ngram beats q27's fused
suffix 653 vs 377 t/s, and I measured **81% of q27 suffix fires pin at
the 12-lane cap** -- truncation, not weak matching. This plan raises the
cap to 16, the natural ceiling of the already-"16-ready" plumbing.

## Honest expectation FIRST

This is a cap raise 12 -> 16, not a path to parity. On file re-emission
the matched span is often longer than 16 (llama accepts ~24-token
drafts), so many fires will simply re-pin at 16. Expected: ~+33% more
accepted tokens per pinned fire, a PARTIAL recovery of the repetition-
regime gap -- not the full 653. True parity needs an unbounded /
variable-width suffix verify, which is a bigger design change (variable
smem in fdmma, no fixed lane count) and is explicitly out of scope here.
If the measured recovery is small, that's a real finding too: it says
the gap is drafting mechanism (unbounded lookup), not verify width.

Novel prose is untouched by construction -- the suffix drafter never
fires there; the MTP ladder path (gate_maxd 4..7) is unchanged. W16
costs nothing on the 224 t/s headline.

## What's already 16-ready (no work)

- Lane-pointer structs P3/CP3/IP3/XQ3: `float*[16]` (kernels.cuh:67-69,
  spec3.cuh:13).
- `Q4Lanes`/`Q8Lanes` in gemv_q4_n/q8_n: `for (i=0; i<16; i++)`
  (kernels.cu:352, 377).
- fdmma verify: `FDMMA_CASE(13..16)` already emitted; comment
  "kernel is 16-ready" (fdmma.cuh).
- Post-collapse lane state: `S_sp/ring_sp[W_PLUMB-1]`, the
  `std::array<float*, W_PLUMB>` lane members, `OUTCOME_INTS = W_PLUMB+2`
  -- all scale off W_PLUMB automatically.
- Graph-capture loop is over `W_MAX` perms; `h_sfx_prop[W_MAX]`,
  `sfx_w` clamp to W_MAX -- all auto.
- Histograms gch/gnh/glf/gla are gate_maxd-DEPTH indexed (max 8), NOT
  suffix-lane indexed -- untouched.

## What actually changes

1. `W_PLUMB 12 -> 16` (engine.cuh:61). New build target `q27-server-w16`
   with `-DQ27_W_MAX=16` (Makefile, mirroring q27-server-w8).
2. gemv_q4_n and gemv_q8_n: add `case 13..16` to both switch lists
   (kernels.cu ~360, ~385). Templates instantiate on reference.
3. Per-lane scratch alloc block: extend the explicit `_L[8..11]`
   sections to `_L[12..15]` (engine.cuh ~586+). Four more lanes; or
   loop-ify while here (the alloc sizes are lane-uniform).
4. Hardcoded lane literals in the hot round path (spec3.cu -- found by
   grep audit, these are the real checklist, NOT auto-scaling):
   - `spec3.cu:696` fdmma dispatch guard `ntok >= 4 && ntok <= 12`
     -> `<= 16` (or `<= W_PLUMB`). Caps the mma verify at 12 today.
   - `spec3.cu:859` `for (k < 11) dr[k] = drafts.p[k]` -> `W_PLUMB-1`.
   - `spec3.cu:861` `for (t < 12) v[t] = verdicts.p[t]` -> `W_PLUMB`.
   - `spec3.cu:868` `for (k = 1; k <= 11)` -> `<= W_PLUMB-1`.
   - `spec3.cu:882` `for (k < 11) outcome[2+k] = dr[k]` -> `W_PLUMB-1`.
   Replace the literals with W_PLUMB-derived expressions so a future
   width change is one constant. Re-grep `\b(11|12|16)\b` in spec3.cu /
   engine.cuh after, filtered against dims (GDN/HEAD/EMBD), to confirm
   none remain in the lane path.

## Costs to measure, not assume

- VRAM: (W_MAX+1) GDN role sets, 13 -> 17 = +4 x ~157MB ~= +630MB per
  engine. Fine on a 32GB 5090; this is a 5090-ONLY build (the 3090 w8
  is already tight, and the 3090's h16 verify path has no 13..16 cases
  -- out of scope).
- Graph zoo: 16 perms x per-width verify graphs (4..16) vs 12 x (4..12)
  ~= 1.8x captured graphs -> more boot time (measure vs the 4.0s W12
  baseline) and more graph VRAM.
- Per-fire compute: a 16-lane verify GEMV costs more than 12 even when
  a fire only accepts <16; the win requires realized acceptance to
  cover the extra width. On file re-emission (drafts >= 16) it should;
  a fire that verifies 16 and accepts 12 paid for 4 wasted lanes. The
  cap-binding re-measure below is the go/no-go.

## Gates

1. **W16-build CLI canonical == a2982c51.** The modulus-relabel
   invariant (perm mod W_MAX; ladder commits <= 8 < 16) predicts the
   ladder path is bitwise-invariant to W_MAX. This is the primary
   correctness gate -- if it fails, the invariant has a hole.
2. test_kernels on the W16 binary: gemv 13..16 and fdmma 13..16
   numerics vs the fd2/reference oracle.
3. Sanitizer (filtered + capped, standing rule) on the widened verify.
4. **Payoff re-measure** (the reason to build it): the file-re-emission
   cap-binding probe on W16 -- does mean accepted-per-fire exceed 12?
   does the n=12 pin spread toward 16? does repetition t/s move from
   377 toward 653? Report the number honestly whichever way it lands.
5. Novel-prose suite unchanged (regression guard -- ladder path).

## Effort / recommendation

Post-lane-collapse this is ~half the 8->12 widening: 4 constants/cases +
one alloc block + the gate campaign. Estimate: a focused session.
Risk is low (invariant-protected, additive, opt-in build). The one real
unknown is gate 4's magnitude -- so build it, measure the recovery, and
let the number decide whether to chase unbounded suffix next. Worst
case it's a cheap, well-gated negative ("width was not the gap").
